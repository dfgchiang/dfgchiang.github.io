GeoNyet

I wish Esri have a better way of building networks of people who are in the same "boat" - I mean people working under similar technical environments and implementations, and handling similar types and amount of data. When a group is simply called Geodatabase, does that mean some tabular operations in a single file geodatabase or managing multiple SQL Server instances on remote virtual machines? Working with those 2 technical areas are very different situations, and I doubt they encounter many similar issues.

People should be not group together simply based on a broad topic such as Python or ArcGIS Server. Different projects can have vastly different scope and scale of application even though people attempt with the same technology. Projects of similar size and goals and objectives have more in common regardless of the API initially applied to it than projects using the same API and platform but having very different objectives and resources.

Serving Lots and Lots of Services

I am curious what is the scale of data and services are people serving and managing out there?

We are serving out thousands of datasets over a couple hundred map services on an Enterprise licensed ArcGIS Server. The Server sits on a virtual server that can have as much resource allocated to it as required, so CPU, RAM and disk spaces are non-issues. Anyway we never see the server performance maxed out. The services can be dynamically added and used in web mapping applications. The map services data reside in SDE on a SQL Server instance. Some of these feature classes are very large or dense datasets that span multiple counties up to the size of the State of California. As we added more and more datasets and services, lately the ArcGIS Server performance are unpredictable at best, and can slow to a crawl when many users are using the web app, but we are having a hard time pinpointing where is the bottleneck. Have others experienced similar problems?

It has been very difficult to get a working answer or online documentation or guide somewhere to explain the technical limitations of Server. What is the maximum number of map services a single Server can serve out as dynamic map services? Approximately how many datasets or amount of data can Server handle? What is the maximum number of Server connections to a SDE database that is recommended? What is an outrageous number of feature classes in a SDE database? Should we spread our data collection across more SDE databases than we already do? Do folks who work in major entities like USGS or EPA have similar questions? I just don't come across many on-premise implementations that is similar to ours as described by people writing in GeoNet.

If it is not a military or industrial secret, Esri ought to publish a list of organizations or groups that are serving out the most amount of data, highest number of services, highest number of datasets, greatest geographic extent of data, highest number of applications, highest number of users, etc. So that other organizations do no feel alone and can contact each other for mutual support.
